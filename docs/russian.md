## Обзор

Kenshi-Online (KenshiMP) — кооперативный мультиплеерный мод для Kenshi, поддерживающий до 16 игроков на одном выделенном сервере. Мод полностью использует нативные системы Kenshi — загрузчик плагинов Ogre3D для инжекции, MyGUI для всего игрового интерфейса и ENet для UDP-сетевого взаимодействия. Никакой инжекции процессов, DLL-инжекции или сторонних оверлеев.

Мод построен как 8 проектов на C++ (плюс 2 тестовых), которые компилируются в клиентскую DLL, исполняемый файл выделенного сервера, исполняемый файл мастер-сервера, лаунчер/инжектор и вспомогательные библиотеки.

Все 13 фаз мультиплеерной реализации полностью функциональны: инжекция, подключение, регистрация сущностей, создание персонажей, синхронизация позиций, бой, инвентарь/экипировка, чат/команды, мировое состояние, отряды/фракции, строения/двери, восстановление при отключении и серверная инфраструктура.

## Требования

- **Kenshi** версии 1.0.59 или новее (протестировано на v1.0.68)
- **Windows 10/11** 64-бит
- **Файлы мода:**
  - `KenshiMP.Core.dll` — клиентский плагин (загружается Ogre)
  - `KenshiMP.Injector.exe` — настраивает Ogre для загрузки плагина, запускает игру
  - `KenshiMP.Server.exe` — выделенный сервер (нужен только для хостинга)
  - `KenshiMP.MasterServer.exe` — реестр браузера серверов (опционально, централизованный)

## Установка и настройка

### Для игроков (подключение к серверу)

1. Поместите `KenshiMP.Core.dll` в директорию игры Kenshi (рядом с `kenshi_x64.exe`)
2. Запустите `KenshiMP.Injector.exe`
3. Укажите **путь к игре**, если он не определён автоматически
4. Введите **имя игрока**
5. Введите **адрес сервера** (IP) и **порт** (по умолчанию: 27800)
6. Нажмите **PLAY**

Инжектор:
- Скопирует `KenshiMP.Core.dll` в директорию игры (если ещё не скопирован)
- Модифицирует `Plugins_x64.cfg`, добавив `Plugin=KenshiMP.Core`
- Запишет конфигурацию подключения в `%APPDATA%/KenshiMP/client.json`
- Запустит Kenshi через Steam (appid 238960)

При запуске Kenshi система плагинов Ogre автоматически загрузит DLL. Мультиплеерный HUD появится в верхней части экрана по центру.

### Для операторов серверов

См. [Как запустить сервер](#как-запустить-сервер) ниже.

## Как играть (Руководство клиента)

### Шаг 1: Запуск через Инжектор

Запустите `KenshiMP.Injector.exe`. Он модифицирует `Plugins_x64.cfg` и запустит Kenshi. Загрузите сохранение или начните новую игру.

После загрузки игры вы увидите строку статуса **KENSHI ONLINE** в верхней части экрана по центру. Панель отладки/загрузки справа показывает прогресс инициализации.

### Шаг 2: Подключение к серверу

Нажмите **F1** для открытия мультиплеерного меню. Оно имеет три вкладки:

- **Quick Connect** — введите IP и порт сервера, затем нажмите Connect
- **Server Browser** — просмотрите серверы, зарегистрированные на мастер-сервере. Нажмите REFRESH для запроса. Выберите сервер и нажмите JOIN
- **Settings** — настройте имя игрока, сервер по умолчанию, автоподключение

После подключения:
- Меню закрывается автоматически
- Строка статуса обновляется: ваше имя, количество игроков, удалённые сущности, пинг
- Персонажи других игроков появляются в вашем мире
- Ваши персонажи транслируются другим игрокам

### Шаг 3: Взаимодействие

- **Двигайтесь обычно** — позиции персонажей синхронизируются каждые 50мс
- **Сражайтесь** — бой авторитетно рассчитывается на сервере и транслируется всем
- **Общайтесь** — нажмите Enter для открытия чата, введите сообщение, нажмите Enter для отправки
- **Команды** — введите `/help` в чат для полного списка команд

### Отключение

Нажмите F1 для открытия меню или введите `/disconnect` в чат. Удалённые персонажи телепортируются под землю и очищаются. При включённом автоподключении клиент попытается переподключиться через 2 секунды.

## Как запустить сервер

### Быстрый старт (локально)

1. Запустите `KenshiMP.Server.exe`
2. Файл конфигурации `server.json` создаётся автоматически при первом запуске
3. Сервер начинает прослушивание на порту **27800 UDP**
4. Сервер автоматически пытается настроить UPnP-маппинг порта
5. Игроки подключаются по вашему IP-адресу (или через браузер серверов при работающем мастер-сервере)

### Настройка сервера (server.json)

```json
{
  "serverName": "Мой сервер Kenshi",
  "port": 27800,
  "maxPlayers": 16,
  "password": "",
  "tickRate": 20,
  "gameSpeed": 1.0,
  "pvpEnabled": true,
  "savePath": "world.kmpsave",
  "masterServer": "127.0.0.1",
  "masterPort": 27801
}
```

| Параметр | Тип | По умолчанию | Описание |
|----------|-----|--------------|----------|
| serverName | строка | "Kenshi Server" | Отображаемое имя в браузере серверов |
| port | uint16 | 27800 | UDP-порт для прослушивания |
| maxPlayers | int | 16 | Максимальное количество игроков (1-16) |
| password | строка | "" | Пароль сервера (пусто = без пароля) |
| tickRate | int | 20 | Частота обновления сервера в Гц |
| gameSpeed | float | 1.0 | Множитель игрового времени (1.0 = обычная) |
| pvpEnabled | bool | true | Разрешить бой между игроками |
| savePath | строка | "world.kmpsave" | Путь к файлу сохранения мира |
| masterServer | строка | "127.0.0.1" | Адрес мастер-сервера для регистрации |
| masterPort | uint16 | 27801 | Порт мастер-сервера |

Свой путь к конфигу: `KenshiMP.Server.exe путь/к/config.json`

### Консольные команды сервера

| Команда | Описание |
|---------|----------|
| `help` | Показать список доступных команд |
| `status` | Показать статус сервера (игроки, сущности, тикрейт, аптайм, память) |
| `players` | Список подключённых игроков с ID, именем, пингом, зоной и количеством сущностей |
| `kick <id>` | Кикнуть игрока по числовому ID |
| `say <сообщение>` | Отправить системное сообщение всем подключённым игрокам |
| `save` | Немедленно сохранить текущее состояние мира на диск |
| `stop` | Сохранить мир и корректно выключить сервер |

### Хостинг на VPS / выделенном сервере

1. **Скопируйте файлы** на сервер:
   - `KenshiMP.Server.exe`
   - `server.json` (опционально — создаётся автоматически)

2. **Откройте порт** UDP 27800 в файрволе:
   ```
   # Windows
   netsh advfirewall firewall add rule name="KenshiMP" dir=in action=allow protocol=UDP localport=27800

   # Linux (при использовании Wine)
   sudo ufw allow 27800/udp
   ```

3. **Отредактируйте server.json** с нужными настройками

4. **Запустите сервер**: `KenshiMP.Server.exe`

5. Игроки подключаются по публичному IP VPS и порту 27800, или находят сервер в браузере

### Сохранение мира

- Сервер автоматически сохраняет мир при выключении (команда `stop` или Ctrl+C)
- Используйте команду `save` для ручного сохранения в любой момент
- При запуске сервер загружает предыдущее состояние мира
- Файл сохранения (`world.kmpsave`) содержит:
  - Позиции, повороты, здоровье (по частям тела) всех сущностей
  - Фракции, шаблоны и владельцев сущностей
  - Время суток, состояние погоды
  - Счётчик следующего ID сущности (предотвращает коллизии при перезапуске)

## Мастер-сервер (Браузер серверов)

`KenshiMP.MasterServer.exe` запускает централизованный реестр серверов на порту **27801 UDP**.

### Как это работает

1. **Игровые серверы регистрируются** при запуске, отправляя `MS_Register` с именем, портом, количеством игроков, лимитом и настройкой PvP
2. **Серверы отправляют heartbeat** каждые 30 секунд через `MS_Heartbeat` (обновлённое количество игроков и время суток)
3. **Клиенты запрашивают** мастер через `MS_QueryList` и получают `MS_ServerList` со всеми зарегистрированными серверами
4. **При выключении** серверы отправляют `MS_Deregister` (или мастер удаляет их по таймауту)

### Переподключение с экспоненциальной задержкой

При разрыве соединения с мастер-сервером игровой сервер использует экспоненциальную задержку:
- Начальная попытка: 5 секунд
- Максимальная задержка: 60 секунд
- Переподключение автоматическое и прозрачное

### Запуск мастер-сервера

```
KenshiMP.MasterServer.exe [порт]
```

Порт по умолчанию: 27801. Мастер-сервер не сохраняет состояние — отслеживает только текущие серверы в памяти.

## Чат и клиентские команды

### Чат

Нажмите **Enter** для открытия ввода чата. Полупрозрачная тёмная панель появляется в нижнем левом углу с 10 видимыми сообщениями. Введите сообщение и нажмите Enter для отправки. Нажмите Escape для отмены.

- **Сообщения игроков** отображаются белым: `[ЧЧ:ММ] ИмяИгрока: сообщение`
- **Системные сообщения** отображаются янтарным (золотым): `[ЧЧ:ММ] [System] сообщение`
- Сообщения исчезают через **30 секунд**
- История чата хранит до **50 записей**

### Клиентские команды

Все команды вводятся в чат с префиксом `/`.

| Команда | Описание |
|---------|----------|
| `/help` | Показать список доступных команд |
| `/connect <ip> [порт]` | Подключиться к серверу (порт по умолчанию: 27800) |
| `/disconnect` | Корректно отключиться от текущего сервера |
| `/tp [имя]` | Телепортироваться к игроку (или к ближайшему, если имя не указано) |
| `/teleport [имя]` | Псевдоним для `/tp` |
| `/time [значение]` | Показать/установить время суток (0.0=полночь, 0.5=полдень, 1.0=полночь) |
| `/pos` | Показать текущую мировую позицию основного персонажа (X, Y, Z) |
| `/position` | Псевдоним для `/pos` |
| `/players` | Список подключённых игроков с их ID |
| `/who` | Псевдоним для `/players` |
| `/status` | Показать количество сущностей, размер очереди создания, загруженные шаблоны |
| `/entities` | Показать количество сущностей (всего, локальных, удалённых, созданных) |
| `/ping` | Показать текущий пинг до сервера в миллисекундах |
| `/debug` | Переключить панель отладки/загрузки |
| `/kick <имя> [причина]` | Кикнуть игрока по имени (только для хоста) |
| `/announce <сообщение>` | Отправить системное сообщение всем (только для хоста) |

## Интерфейс HUD

HUD использует нативный рендеринг **MyGUI** Kenshi — без внешних оверлеев. Все виджеты используют собственные скины Kenshi (`Kenshi_FloatingPanelSkin`, `Kenshi_TextboxStandardText`, `Kenshi_TextboxPaintedText`).

### Строка статуса (верх по центру)

Всегда видна при активном HUD. Показывает:
- **Не подключён:** `KENSHI ONLINE | NOT CONNECTED | F1 = Menu | Enter = Chat`
- **Подключён:** `KENSHI ONLINE | CONNECTED as ИмяИгрока | 3 players online | 12 remote | 45ms`

Полупрозрачная тёмная панель с янтарным текстом.

### Панель чата (нижний левый угол)

Полупрозрачная тёмная фоновая панель (альфа 0.7), появляющаяся при наличии сообщений или активном вводе. Содержит:
- **10 строк чата** — последние сообщения внизу
- **Строка ввода** — видна при активном чате, янтарный текст с приглашением `> ` и курсором `_`
- Системные сообщения окрашены **янтарным** (0.9, 0.55, 0.1), чат игроков — **белым** (0.9, 0.9, 0.9)
- Все сообщения с меткой времени `[ЧЧ:ММ]`

### Список игроков (верхний правый угол)

Переключается клавишей **Tab**. Полупрозрачная панель:
- Заголовок: "PLAYERS ONLINE"
- До **8 строк игроков**
- Локальный игрок: `ВашеИмя (You)` или `ВашеИмя (You) [HOST]`
- Удалённые игроки: `ИмяИгрока [IN WORLD]` или `ИмяИгрока [loading]`

### Панель отладки/загрузки (правая сторона)

Переключается клавишей **Insert**. Видна по умолчанию во время загрузки, автоматически скрывается через 15 секунд после загрузки игры (если не подключён). Показывает:
- Заголовок: "KENSHI ONLINE"
- До **20 строк лога** в формате `[ТЕГ] сообщение`
- Теги: INIT, HOOK, SCAN, NET, GAME, OK, ERR, WARN
- Подсказка внизу: `Insert = toggle | F1 = menu | Enter = chat`

## Справочник сетевого протокола

### Транспорт

- **Библиотека:** ENet (надёжный UDP)
- **Порт по умолчанию:** 27800 (игра), 27801 (мастер-сервер)
- **Каналов:** 3

| Канал | Название | Надёжность | Порядок | Использование |
|-------|----------|------------|---------|---------------|
| 0 | Reliable Ordered | Надёжный | Упорядоченный | Подключение, сущности, отряды, фракции, строения, чат, админ |
| 1 | Reliable Unordered | Надёжный | Неупорядоченный | Бой, статы, здоровье, инвентарь, экипировка, торговля |
| 2 | Unreliable Sequenced | Ненадёжный | Последовательный | Обновления позиций (движение) |

### Формат пакета

Каждый пакет начинается с 8-байтового заголовка:

```
PacketHeader (8 байт, упакованный):
  [0] MessageType type     (uint8)   — Тип сообщения
  [1] uint8_t flags        (uint8)   — Бит 0: сжатие
  [2] uint16_t sequence    (uint16)  — Порядковый номер
  [4] uint32_t timestamp   (uint32)  — Серверный тик
```

### Лимиты пропускной способности

- **Исходящий:** 128 КБ/с на клиента
- **Входящий:** 256 КБ/с на клиента

### Тайминги

- **Тикрейт:** 20 Гц (интервал 50мс)
- **Таймаут подключения:** 5000мс
- **Интервал keepalive:** 1000мс
- **Таймаут бездействия:** 10000мс

## Архитектура и структура проекта

```
[Клиент Kenshi 1]  <--ENet UDP-->  [Выделенный сервер]  <--ENet UDP-->  [Клиент Kenshi 2]
       |                                    |                                    |
 KenshiMP.Core.dll              KenshiMP.Server.exe               KenshiMP.Core.dll
 (хуки + нативный UI)          (авторитет + ретрансляция)         (хуки + нативный UI)
                                           |
                                KenshiMP.MasterServer.exe
                                (реестр браузера серверов, порт 27801)
```

### Проекты (10 целей сборки)

| Проект | Тип | Описание |

## Система инжекции и плагинов

KenshiMP использует **систему плагинов Ogre3D** — тот же механизм, который Kenshi использует для загрузки собственных плагинов рендеринга. Это неинвазивно и обратимо.

### Как это работает

1. **Инжектор** модифицирует `Plugins_x64.cfg` в директории Kenshi, добавляя строку:
   ```
   Plugin=KenshiMP.Core
   ```
2. **Kenshi запускается** обычным способом через Steam
3. **Ogre3D** читает `Plugins_x64.cfg` при инициализации движка и загружает `KenshiMP.Core.dll` как стандартный плагин Ogre
4. **dllmain.cpp** реализует интерфейс плагина Ogre (`installPlugin` / `uninstallPlugin`)
5. **Core::Initialize()** запускает: сканер паттернов, 14 модулей хуков, ENet-клиент, мост MyGUI, реестр команд

### Удаление

Инжектор может удалить строку плагина из `Plugins_x64.cfg`, восстановив ванильный Kenshi. Файлы данных игры не модифицируются.

### Почему не DLL-инжекция?

Традиционная DLL-инжекция (CreateRemoteThread, LoadLibrary):
- Определяется антивирусами
- Нестабильна между обновлениями Windows
- Требует запущенной игры

Подход через плагин Ogre:
- Чистый и нативный для игры (как все плагины рендеринга Kenshi)
- Загружается при инициализации движка (до любой игровой логики)
- Не определяется антивирусами
- Тривиально обратим

## Конвейер создания сущностей

Конвейер создания — самая критичная и деликатная часть мода. Удалённые персонажи должны создаваться через собственную фабрику Kenshi во избежание повреждения памяти.

### Единственный безопасный метод: In-Place Replay

Когда игра естественно создаёт NPC (через `CharacterCreate`), хук в `entity_hooks.cpp`:

1. **Перехватывает** вызов фабрики
2. **Проверяет** очередь создания на наличие ожидающих удалённых сущностей
3. **Воспроизводит** вызов фабрики по **тому же адресу стека** с изменёнными параметрами
4. Новый персонаж создаётся через тот же путь кода, что и естественный NPC
5. После создания: применяются **WriteName**, **WriteFaction**, **WritePosition**

### Лимиты и безопасность

- **Лимит создания на игрока:** `MAX_SPAWNS_PER_PLAYER = 8` (предотвращает краш от потока NPC)
- **Максимум воспроизведений за один вызов хука:** 3 (предотвращает зацикливание)
- **Максимум попыток создания:** 200 (~10 секунд при 20 Гц)
- **Запасной вариант:** `SpawnCharacterDirect` после 5-секундного таймаута (работает, но персонажи могут дестабилизироваться через 10-20 секунд)

### Загрузка фракции (Bootstrap)

Когда указатель фракции игрока недоступен через `PlayerBase` (часто на Steam), `entity_hooks` захватывает фракцию от первого созданного персонажа. Core затем пересканирует существующих персонажей через `RequestEntityRescan()` для применения правильной фракции к уже зарегистрированным сущностям.

## Синхронизация позиций и интерполяция

### Исходящий поток (Локальный -> Сервер -> Другие)

1. **BackgroundReadEntities** сканирует всех локальных персонажей каждый тик
2. Позиции сравниваются с последними отправленными значениями по порогам:
   - Позиция: изменение > 0.1 метра
   - Поворот: изменение > 0.01
3. Изменённые позиции группируются в пакеты `C2S_PositionUpdate`
4. Пакеты включают: entityId, позицию (3 float), сжатый кватернион (наименьшие три), состояние анимации, скорость движения (0-255, отображается в 0-15 м/с), флаги (бег/скрытность/бой)
5. Отправляются по **каналу 2** (ненадёжный последовательный) для минимальной задержки

### Ретрансляция сервера

Сервер получает обновления позиций, сохраняет, обновляет зоны и ретранслирует другим клиентам с фильтрацией по **зонной системе интересов** (только игроки в соседних зонах получают обновления).

### Входящий поток (Другие -> Локальное отображение)

1. **Система интерполяции** получает снимки с метками времени
2. Каждая сущность имеет **кольцевой буфер из 8 снимков**
3. **Адаптивная оценка джиттера** регулирует задержку интерполяции:
   - Фактор сглаживания EMA: 0.1
   - 20мс джиттер -> 50мс задержка (минимум)
   - 80мс джиттер -> 200мс задержка (максимум)
4. **Интерполяция сплайнами Эрмита** между двумя ограничивающими снимками
5. **Экстраполяция dead reckoning** до 250мс после последнего снимка (используя скорость)
6. **Коррекция рывков** для больших ошибок позиции:
   - < 5 метров: плавное смешивание (игнорируется)
   - 5-50 метров: коррекция смешиванием за 150мс
   - > 50 метров: мгновенная телепортация
7. Интерполированные позиции записываются в игровых персонажей через `CharacterAccessor::WritePosition()`

### Двойная буферизация данных кадра

Для предотвращения конкуренции между игровым потоком и сетевыми воркерами:
- Воркеры записывают состояния сущностей в буфер A
- Игровой поток читает из буфера B
- Атомарная замена каждый тик

## Синхронизация боя

Бой является **серверно-авторитетным** — сервер рассчитывает весь урон.

### Поток

1. Локальный персонаж атакует -> `combat_hooks` перехватывает `ApplyDamage` или `StartAttack`
2. Клиент отправляет `C2S_AttackIntent` (attackerId, targetId, attackType)
3. Сервер рассчитывает бой:
   - Случайный выбор части тела (Голова, Грудь, Живот, Левая рука, Правая рука, Левая нога, Правая нога)
   - Расчёт урона (режущий, дробящий, пронзающий)
   - Оценка шанса блока
4. Сервер рассылает `S2C_CombatHit` всем релевантным клиентам
5. Клиенты применяют урон через оригинальную функцию `ApplyDamage` (или прямая запись в память как запасной вариант)
6. При пересечении порога здоровья:
   - `S2C_CombatKO` для нокаута
   - `S2C_CombatDeath` для смерти

### Синхронизация боевой стойки

Игроки могут менять стойку (пассивная/оборонительная/агрессивная/удержание) через `C2S_CombatStance`, рассылается всем.

## Синхронизация отрядов и фракций

### Отряды

- `C2S_SquadAddMember` -> сервер валидирует -> `S2C_SquadMemberUpdate` (действие: добавлен/удалён)
- Хуки отрядов подавляются при сохранении/загрузке

### Фракции

- Изменения отношений фракций перехватываются `faction_hooks`
- `C2S_FactionRelation` (factionIdA, factionIdB, отношение float от -100 до +100, causerEntityId)
- Сервер валидирует и рассылает `S2C_FactionRelation`
- **Защита от петли обратной связи:** `faction_hooks::SetServerSourced(true/false)` предотвращает повторную отправку хуком изменения, только что применённого от сервера

## Синхронизация строений и дверей

### Размещение строений

1. `building_hooks` перехватывает размещение -> `C2S_BuildRequest` (templateId, позиция, поворот)
2. Сервер назначает ID сущности -> рассылка `S2C_BuildPlaced` (призрачная сущность)
3. Прогресс строительства: `S2C_BuildProgress` (0.0 до 1.0)
4. Сервер валидирует владение для демонтажа (`C2S_BuildDismantle`) и ремонта (`C2S_BuildRepair`)

### Состояние дверей

- `C2S_DoorInteract` (entityId, actorEntityId, действие: открыть/закрыть/замкнуть/отомкнуть)
- Сервер рассылает `S2C_DoorState` (entityId, состояние: закрыта/открыта/заперта/сломана)
- Состояние двери записывается через указатель функциональности строения (`functionality+0x10`)

### Ограничение

Визуальное создание строений (создание игровых объектов для удалённых строений) пока не реализовано — требуется реверс-инжиниринг фабрики строений. Призрачные сущности отслеживаются в реестре.

## Синхронизация времени и мирового состояния

### Система времени

Время суток хранится в объекте **TimeManager** (НЕ в GameWorld). Модуль `time_hooks` захватывает указатель TimeManager при вызове функции `TimeUpdate`.

- **Чтение:** `time_hooks::GetTimeOfDay()` читает `TimeManager+0x08` (float 0.0-1.0, 0=полночь, 0.5=полдень)
- **Запись:** `time_hooks::WriteTimeOfDay(value)` записывает `TimeManager+0x08`
- **Скорость игры:** `TimeManager+0x10` (float множитель)

### Серверная синхронизация времени

- Сервер периодически рассылает `S2C_TimeSync` (serverTick, timeOfDay, weatherState, gameSpeed)
- Клиент получает -> `time_hooks::SetServerTime()` записывает время
- Хост освобождён от перезаписи времени (хост ведёт авторитетные часы)

## Зонная система интересов

Мир разделён на зоны по **750 метров**. У каждого игрока радиус интереса **1 зона** (сетка 3x3, или 2250м x 2250м видимая область).

### Как это работает

1. Сервер отслеживает текущую зону каждого игрока (вычисляется из позиции)
2. При рассылке обновлений сущностей сервер проверяет: находится ли сущность в соседней зоне получателя?
3. Только сущности в пределах сетки 3x3 отправляются
4. Переходы зон запускают переоценку видимых сущностей для каждого игрока

### Константы

- `KMP_ZONE_SIZE = 750.0` метров на зону
- `KMP_INTEREST_RADIUS = 1` (±1 зона от игрока)
- `KMP_AUTHORITY_HYSTERESIS = 64.0` единиц до передачи авторитета зоны
- `KMP_MAX_ENTITIES_PER_ZONE = 512`
- `KMP_MAX_SYNC_ENTITIES = 2048` всего на клиента

## Отключение и восстановление

### Обнаружение отключения

Клиент определяет отключение, когда `Core::IsConnected()` возвращает true, но `NetworkClient::IsConnected()` — false.

### Поток очистки

1. **Телепортация удалённых сущностей под землю** (`y = -10000`) — должна произойти ДО очистки реестра (который стирает указатели на игровые объекты)
2. **SetConnected(false)** запускает полную очистку:
   - Очистка реестра сущностей (удалённые сущности)
   - Очистка системы интерполяции
   - Сброс контроллера игрока
   - Сброс состояния подключения
3. **Системное сообщение:** "Disconnected from server."
4. **Автопереподключение:** при включённом автоподключении попытка через 2 секунды

## Безопасность сервера

Выделенный сервер реализует множественные меры безопасности:

| Функция | Описание |
|---------|----------|
| **Предотвращение дублей** | Только одно подключение на игрока; дубликаты отклоняются |
| **Валидация владения** | Игроки могут контролировать только собственные сущности |
| **Проверка границ зон** | Обновления позиций валидируются против границ карты |
| **Валидация торговли** | Обе стороны должны присутствовать с нужными предметами/средствами |
| **Валидация фракций** | Игроки не могут устанавливать произвольные отношения |
| **Лимит создания** | `MAX_SPAWNS_PER_PLAYER = 8` предотвращает спам сущностями |
| **Лимит сущностей** | Проверка `WouldExceedLimit()` перед созданием |
| **Лимит длины строк** | `PacketReader::ReadString` отклоняет строки более 1024 байт |
| **Проверка версии** | Handshake валидирует совпадение `KMP_PROTOCOL_VERSION` |
| **Ограничение скорости** | Менеджер игроков отслеживает частоту подключений |
| **Таймаут keepalive** | 10-секундный таймаут бездействия отключает нереагирующих |

## Устранение неполадок

| Проблема | Решение |
|----------|---------|
| HUD не появляется | Убедитесь, что Kenshi запущен через Инжектор. Проверьте наличие `Plugin=KenshiMP.Core` в `Plugins_x64.cfg` |
| «Connection failed» | Проверьте IP/порт сервера. Убедитесь, что UDP-порт 27800 открыт. Проверьте UPnP в логе сервера |
| Персонажи не двигаются | Нажмите Insert для панели отладки. Проверьте, что `SetPosition` показывает "resolved". Если нет — сканер паттернов не нашёл функцию |
| Удалённые персонажи не появляются | Проверьте лог отладки на статус очереди создания. Игра должна естественно создать NPC для работы in-place replay. Перемещайтесь, чтобы инициировать появление NPC |
| Высокий пинг / лаг | Сервер должен быть географически близко. Проверьте тикрейт и сетевые условия |
| Сервер не запускается | Проверьте, не занят ли порт 27800 (`netstat -an | find "27800"`). Попробуйте другой порт |
| Краш игры при запуске | Проверьте `KenshiOnline.log`. Частая причина: сбой сканера паттернов. Для лучшей совместимости используйте GOG v1.0.68 |
| Браузер серверов пуст | Убедитесь, что мастер-сервер запущен. Проверьте, что `masterServer` в конфиге игрового сервера указывает на правильный адрес |
| Чат не работает | Убедитесь, что нажимаете Enter (не Return на numpad). Проверьте установку хуков ввода в логе |
| Не работают kick/announce | Эти команды только для хоста. Первый подключившийся — хост |

### Файлы логов

- **Клиент:** `KenshiOnline.log` (в директории Kenshi)
- **Сервер:** `KenshiOnline_Server.log` (в директории сервера)
- **Отладочный вывод:** сообщения `OutputDebugString` с префиксом `KMP:` (видны в DebugView или отладчике VS)

## Известные ограничения

- **Визуальное создание строений** — Строения отслеживаются на сервере и в реестре, но игровые объекты не создаются для удалённых игроков. Требуется RE фабрики строений для in-place replay
- **ИИ NPC** — Синхронизация NPC односторонняя. Ваши NPC видны другим, но ИИ работает независимо. ИИ подавляется на удалённых персонажах для предотвращения конфликтов
- **Сохранения игры** — Каждый игрок использует своё сохранение Kenshi. Сохранения не синхронизируются
- **RVA синглтонов Steam** — Жёсткие адреса PlayerBase и GameWorld для GOG v1.0.68. Steam имеет другой макет `.data`. Запасной вариант через string xref работает, но может не сработать на некоторых сборках Steam
